# SG-4 — the MCP SDK v2 break, and why zing was on track to ship into it

**Lane B, 2026-07-20.** Sources fetched live this cycle; every claim
below is checkable from them.

## The watch-item fired, eight days early

I filed "MCP spec 2026-07-28 goes final; **SDK support = trigger**"
(SG-4, 2026-07-19) and set out this cycle to re-verify its premise
rather than wait for the date. The premise held — the spec still lands
**2026-07-28** — but the trigger had already fired: **beta SDKs for the
release candidate shipped 2026-06-29**, Python among them
(`mcp==2.0.0b1`).

Waiting for the calendar date would have been waiting for the wrong
event. The date the spec is *published* is not the date the ecosystem
changes under us; the SDK release is.

## What actually breaks

The v2 migration guide is unambiguous:

> "The `FastMCP` class has been renamed to `MCPServer`."
> "All submodules under `mcp.server.fastmcp.*` are now under
> `mcp.server.mcpserver.*` with the same structure."

`mcp.server.fastmcp` does not exist in v2. Zing imports it in four
places (`FastMCP`, `prompts.Prompt`, `utilities.types.Image` twice).

## The defect this created on our side

`pyproject.toml` declared `mcp = ["mcp>=1.2"]` — **unbounded**. Once
`2.0.0` leaves beta, a fresh `pip install "myzing[mcp]"` resolves to v2
and every import site fails. New users only; existing installs are
pinned by their lockfile or already-resolved environment. So the failure
would have arrived silently, on a date nobody chose, for exactly the
people with the least context to diagnose it.

Worse, both guards tested `import mcp`, which **succeeds** on v2 (the
top-level package is still there). Simulated it:

```
--- `import mcp` guard result ---
  guard PASSES: top-level mcp imports fine (version 2.0.0)
--- what happens when the server is actually built ---
  ImportError: No module named 'mcp.server.fastmcp'
```

A bare traceback: no cause, no version, no fix.

## What was done

1. **`mcp>=1.2,<2`** in both the `mcp` and `dev` extras — the SDK's own
   recommendation to library maintainers, with the reason inline so the
   bound is not read as cargo-culting.
2. **`mcp_sdk_defect()`** now separates ABSENT from INCOMPATIBLE and
   names the installed version, where the API moved, and the pin that
   fixes it. Telling a v2 user to "install the SDK" would have been
   wrong — it *is* installed.
3. **Two gates**: the upper bound cannot be dropped silently, and the
   fix string must name the same constraint `pyproject.toml` declares
   (a guard that recommends an unsupported pin is worse than none).

## Honest limits

- I did **not** install `2.0.0b1` and run the suite against it. The
  break is established from the migration guide plus our import sites,
  and simulated at the import layer — not observed end-to-end.
- No stable `2.0.0` date is published beyond "with the spec". The bound
  is protection, not a prediction.
- The v2 API is not evaluated here. Porting is a real piece of work
  (rename, four imports, re-verify 19 tools) and deserves its own item,
  not a dependency bump. Both blog posts stress that nothing breaks for
  existing servers on 2026-07-28, so there is no urgency to port — only
  urgency to not *drift* into it.

## Next trigger

`mcp 2.0.0` leaving beta. At that point the port becomes a scheduled
decision. Until then the bound holds and the error is honest.

Sources: the MCP blog's release-candidate and beta-SDK posts, the Python
SDK v2 migration guide, and PyPI's `mcp` release metadata (latest stable
`1.28.1`, 2026-06-26 — which is also what this repo runs).
