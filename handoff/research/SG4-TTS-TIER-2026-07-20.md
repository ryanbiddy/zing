# SG-4 scan: the optional TTS tier (the gap my own #248 left)

Scope: my 2026-07-19 stack-currency scan checked scenedetect,
faster-whisper, rapidocr, the mcp SDK and yt-dlp — and skipped the
OPTIONAL tier I own. This closes that gap. It found a live defect.

## Upstream state

| Dependency | Latest | Released | Python | Note |
|---|---|---|---|---|
| kokoro-onnx | 0.5.0 | 2026-01-30 | **>=3.10, <3.14** | our local default voice engine |
| onnxruntime | 1.27.0 | 2026-06-15 | >=3.11 | MIT; kokoro's runtime |

## THE FINDING: doctor reported voiceover ready on a Python that cannot run it

`zing` declares `requires-python = ">=3.10"`. `kokoro-onnx` caps at
`<3.14`. This box runs **Python 3.14.6**, so the package is not
installable — yet `tts_status()` reported `ready: True` with detail
"model files present", and `zing doctor` printed a healthy TTS line.

The bug is the shape audit #201 and D-11 both had: **readiness was
tested through a PROXY (model files on disk) rather than the capability
(can the synthesis runtime be imported).** A user on 3.14 got "ready"
and then a voiceover failure at RENDER time — the most expensive place
to learn it.

Fixed: readiness now requires model files AND `kokoro_onnx` importable.
The detail names the Python version and the two real ways forward
(supported Python, or the ElevenLabs provider). The doctor FIX text is
branch-aware — prescribing a model download to someone whose Python
cannot run the engine is exactly the re-prescribed-fix dead end D-13
and the review's P1-1 punished.

Live after the fix:

```
[ absent] tts (optional)
   selected TTS 'kokoro' not ready: model files present but kokoro-onnx
   is not importable in this Python (3.14) — install it ..., or if your
   Python is too new for it, use the elevenlabs provider ...
   fix: install the render extras so kokoro-onnx is importable ...
```

## Why the existing skip did not catch it

`tests/test_eval_voiceover.py` skips with "set ZING_REQUIRE_KOKORO=1 in
the isolated Python 3.10-3.13 gate" — the version ceiling was KNOWN and
handled for the eval gate. Nobody carried it into the readiness REPORT,
which is the surface a user actually reads. A known constraint that
lives only in a test skip is a constraint the product does not know.

## Not changed

The `<3.14` ceiling is upstream's, not ours. Zing supporting 3.10+
while an OPTIONAL provider supports 3.10–3.13 is legitimate — the
optional tier degrading honestly is the whole point of the tier. No
dependency floor changes; no pin added.
