# Suite smoke in CI

The `suite-smoke` workflow runs the three-product family scenario on every
Zing pull request and main push, once a day at 06:23 UTC, and on manual
dispatch. It uses `deterministic_ci`: a four-second video generated by FFmpeg
inside the runner, not a platform URL.

The job checks out the tested Zing revision plus current Uoink and Writer
`main`, installs their runtime dependencies, and launches the real Uoink and
Writer loopback services. It then connects directly to all three MCP servers
and drives the same contract clients and assertions as the local family gate:

1. validate leases, manifests, health responses, peer states, and direct MCP
   identities;
2. seed one isolated Uoink item with retained fixture media;
3. make Zing verify and study that kept file without refetching it;
4. make Writer retrieve the same corpus item, save and critique a script, run
   its local Voice DNA scan, and export a versioned shot-list file;
5. import that file into Zing, assemble and render a measured draft, account
   for engagement events, stop Uoink, and prove Writer and Zing fail calm;
6. stop every process and independently evaluate the path-free record.

The workflow uploads `suite-smoke.json` and `suite-smoke-eval.json` for 14
days. The first file records the three source revisions, installed versions,
step timings, contract versions, stable references, artifact hashes,
engagement receipts, peer states, and cleanup result. The second file
recomputes 17 assertions from that record instead of trusting its claimed
result. If the launcher fails before it can write the record, the artifact
contains `suite-smoke-failure.json` with the structured launcher error. A
missing record or failure artifact is itself an artifact-step error, so an
early failure cannot look green.

## What the scheduled gate does not prove

Scheduled CI does not run `real_capture`. It does not prove that a current
YouTube, TikTok, Instagram, or X URL can be fetched; that platform captions,
cookies, bot checks, and extractor behavior still work; or that Uoink creates
a retained-media sidecar through its live capture route. It also does not
download an AI model, call an AI or speech provider, exercise desktop GUIs or
browser extensions, or validate a packaged installer.

Those are deliberate limits. The closing S6 real-capture record remains the
one-machine proof for the network capture boundary. Product-specific
Windows, macOS, clean-install, extractor, and packaging gates keep their own
scope. This Ubuntu workflow protects the deterministic cross-product contract
path on every change and every day.

Neither mode may publish, post, submit, invoke a delivery surface, or spend.
The runner removes provider credentials, forces model caches offline, refuses
occupied suite ports, records no credential value or private absolute path,
and stops every process it starts.

## Run it locally

All three repositories must be clean Git checkouts, and ports 5179 and 5181
must be free:

```bash
python -m tools.eval.run_suite_smoke \
  --mode deterministic_ci \
  --uoink-repo <uoink-checkout> \
  --writer-repo <writer-checkout> \
  --zing-repo . \
  --output suite-smoke.json

python -m tools.eval.suite_smoke \
  suite-smoke.json \
  --report suite-smoke-eval.json
```

Use `real_capture` only as an explicit local gate with a supported source URL.
It is not part of scheduled CI.
